# Android 设备实时日志采集方案

## 目标

- 单次采集会话时长以小时计（首个场景约 5 小时），全量抓取指定 Android 设备的 logcat
- 支持多种 Android 设备，运行时由用户选择目标设备，不将任何具体机型或序列号写入方案
- 采集过程中 USB 或网络连接可能中断，恢复后自动继续，无需人工干预
- 不丢失重连后的新日志，不重复灌入设备 buffer 里的历史日志
- 采集结果实时写入本机磁盘，供事后离线分析

## 使用形态

- 通过 `adb` 与已连接的 Android 设备通信，不假设厂商与 Android 版本
- 启动服务时，工具枚举当前所有可用 adb 设备并交由用户选择：
  - 单设备：直接采集
  - 多设备：列出序列号、型号，等待用户挑选
  - 通过参数直接指定序列号跳过交互
- 每次运行只针对一台设备。若需并采多台，允许多次启动工具进程，各自独立进程与产物

## 需求拆解

- 全量：覆盖 main / system / crash / events / radio / security 全部可读 buffer（kernel 需 root 权限，本方案不覆盖）
- 实时：日志逐行落盘，允许外部工具在采集过程中增量读取，本 project 自身不提供查看界面
- 断线自愈：连接中断导致 adb 进程退出时，工具自身不退出，等待设备回来后自动续采
- 防倒灌：重连时不能把设备 buffer 里已经写过的历史行再次写入本机文件
- 抖动缓冲：短时断连不应丢日志
- 可观测：产物本身透露状态（心跳文件 mtime、事件文件末尾），用户借助任意工具即可判断采集是否活着、上次断连发生在什么时候
- 优雅退出:手动中止时不留残留的 adb 子进程

## 稳定性设计

- **设备选择**：启动时基于 `adb devices` 的当前结果动态生成候选，避免把设备信息硬编码
- **断线重连**：由外层守护循环驱动，adb 进程退出后阻塞等待目标设备回连，回连后重新拉起 logcat 采集，直到人工中止
- **防倒灌**：每次拉起 logcat 时携带「仅取当前时刻之后」的时间过滤条件，绕过设备 buffer 里的历史行
- **抖动缓冲**：会话开始时对所选设备扩大 logcat buffer 到 8 MiB，给短断连留出保底容量；扩容失败则降级不扩容
- **实时落盘**：采集流不做聚合缓冲，逐行写入本机文件；终端不回显 logcat 内容，用户直接读文件即可
- **事件旁路**：工具自身的启动、断连、重连、退出事件写入独立文件，与业务日志分离，事后回溯断连时间点方便
- **心跳指标**：每收到一行日志就更新一个心跳文件的 mtime，用于健康检查判活

## 技术栈

- **语言**：Rust（stable 版本，2021 edition 及以上）
- **异步运行时**：tokio，用于统一处理子进程 IO、定时器、信号、内部通道等多路事件
- **产物形态**：单个静态可执行二进制，目标平台 macOS ARM64
- **外部依赖**：仅依赖用户本机已安装的 `adb`；除此之外无任何运行时环境要求（无需 Python / Node / uv / 虚拟环境）
- **关键机制到需求的映射**：
  - **子进程管理**：`tokio::process::Command` 拉起 `adb`，`Child::kill_on_drop(true)` 保证进程 drop 时自动 kill，杜绝退出残留
  - **IO 分离**：`stdout` 走业务日志落盘，`stderr` 用于分类识别断连原因（`device offline` / `not found` / `closed`），驱动不同的退避策略
  - **断线重连循环**：外层 loop 内每次重启 `adb logcat` 前重算 `-T` 时间戳，避免历史行倒灌
  - **状态机**：以 `enum` 建模 `WaitingDevice` / `Streaming` / `Backoff` / `Draining` / `Stopped` 等状态，转移在编译期覆盖
  - **背压**：读线程与写盘/心跳线程之间用有界 `tokio::sync::mpsc::channel`，写盘阻塞时向上传导，不丢日志
  - **心跳节流**：`tokio::time::interval` 每秒 tick 一次更新心跳文件 mtime，避免逐行 syscall
  - **优雅退出**：`tokio::signal::ctrl_c` 触发 → 停止拉起 → drain 通道 → flush 写盘 → drop child 自动 kill
- **允许的第三方 crate 范围**：主流成熟库（如 `tokio`、`clap`、`serde` / `serde_json`、`time` / `chrono`、`anyhow` / `thiserror`），不引入未维护或小众依赖
- **分发方式**：`cargo build --release` 产出二进制，用户拷入个人 PATH（例如 `~/bin/`）即可运行；升级即替换该文件
- **macOS 兼容性说明**：从他人机器传入的二进制首次运行可能被 Gatekeeper 拦截，需清除 quarantine 属性或做签名/公证；用户本地自行 build 无此问题

## 目录与产物

- 可执行文件位置：由用户自行决定，建议放入个人 PATH（例如 `~/bin/`）
- 输出根目录：`~/.config/jj-android-logs/`（首次运行时自动创建）
- 按设备与会话隔离产物，命名建议：
  - `<serial>/readme.md`：该设备的详细描述档案，见下节
  - `<serial>/session-YYYYMMDD-HHMMSS.log`：本次会话完整日志，逐行追加实时写入
  - `<serial>/session-YYYYMMDD-HHMMSS.events.log`：工具自身事件（启动、断连、重连、退出）
  - `<serial>/.heartbeat`：最近一次收到日志行的时间戳文件，供健康检查
  - `<serial>/session.pid`：当前会话进程 ID，用于检测重复启动
- 多设备同时采集时，各自序列号目录互不干扰

## 设备档案 readme.md

- 生成时机：工具选中设备后、正式开始采集前检查该文件；不存在时才生成一次，已存在则跳过（同一设备后续会话不重复写入，避免覆盖历史手改内容）
- 目的：让事后翻看日志的人无需再连设备，即可对齐日志上下文
- 记录范围（凡是能通过 `adb` 拉到的都尽量收全）：
  - **身份**：序列号、厂商、品牌、型号、产品名、设备代号
  - **系统**：Android 发行版本、API level、Build 指纹、Build 类型（user/userdebug）、安全补丁级别、内核版本
  - **硬件**：CPU 架构与支持的 ABI、CPU 核数与频率、总内存、屏幕分辨率与密度、电池信息
  - **网络**：连接方式（USB / TCP，含 TCP 端口）、WiFi/蓝牙 MAC（若可读）、当前 IP
  - **存储**：`/data` 与 `/sdcard` 可用空间
  - **adb 与调试**：授权状态、是否 root、adb 版本、adbd 运行用户
  - **logcat 能力**：各 buffer 当前实际大小、是否成功扩容
  - **时区与时间**：设备时区、设备当前时间与本机时间差
  - **原始快照**：完整的 `getprop` 输出、`dumpsys` 关键分片，作为兜底附在文末，供事后翻查
- 内容格式：Markdown，能被 GitHub / 编辑器直接渲染阅读
- 失败降级：任一项获取失败不阻塞采集，档案里对应项标注「不可用」并继续

## 运行形态

- CLI 前台阻塞运行，占用当前终端，Ctrl-C 结束；本工具不内建后台守护，需常驻由用户自行 `nohup ... &` / `screen` / `tmux` 承担
- 终端 MUST NOT 刷屏 logcat 正文，只输出下述三类结构化行；日志正文全量落盘，用户自行 `tail` / `rg` 产物目录查看
  - **启动摘要**（一次性）：工具版本 / pid / 会话开始时间；设备身份（序列号 / 厂商 / 型号 / Android 版本 / API level / 连接方式与端口）；logcat 目标 buffer 与扩容结果；`session.log`、`session.events.log`、`.heartbeat` 三个产物文件的绝对路径；`readme.md` 是否本轮新生成
  - **周期心跳**（30s tick）：已运行时长 / 累计行数 / 累计字节（human-readable）/ 最近窗口速率（lines/s、bytes/s）/ 最近一行日志的设备时间戳 / 当前状态机状态 / 累计断线次数 / 累计重连次数
  - **事件即时行**（事件驱动，与 `events.log` 内容一致）：断线（含 stderr 原因分类）/ 重连成功（含新 `-T` 起点）/ buffer 扩容失败 / 设备 unauthorized / 优雅退出
- 心跳与事件行走 stdout；MUST 单行结构化（`key=value` 或紧凑 JSON），便于 `tee` 到文件或被上游脚本消费
- 单设备单会话：同一台设备只允许存在一个采集进程，避免两条 logcat 相互干扰；启动时检测已有 pid 文件与目标序列号，冲突则拒绝启动

## 前置一次性设置

- 目标 Android 设备开启开发者选项与 USB 调试（或 WiFi 调试）
- 首次连接后在设备端确认 adb 授权，`adb devices` 中显示 `device` 而非 `unauthorized`
- 本机安装 `adb`（Homebrew `android-platform-tools` 或 Android SDK Platform Tools）
- 具体设备是否支持扩大 logcat buffer 视厂商实现而定，方案不假设一定成功，失败时降级为不扩容继续采集

## 明确不做的事

- 不提供日志查看能力：不内建 UI、tail、grep、检索或健康监控命令，产物为普通文本文件与时间戳文件，用户自行选用外部工具查看与分析
- 不做设备发现之外的自动配对：设备连线、授权由用户完成
- 不引入长期守护（如 `launchd`）：以小时计的单次任务，前台或 `nohup` 起足够
- 不做日志切片轮转：单会话体量在小时级可控，避免引入额外依赖
- 不上结构化采集（`vector` / `fluent-bit`）：无检索、告警、多设备聚合需求
- 不做 tag 过滤：需求是「全量」，过滤留给事后用 `rg` 做

## 验证标准

- 多设备场景：连入两台 Android 设备，启动时能列出两者，用户选择其中任一台后仅采集所选设备
- 断线重连：拔线 10 秒后重新插上，`events.log` 应先后出现「logcat 退出」和「logcat 重启」两条记录，业务日志文件继续增长，且新增内容不含重连前已写入过的行
- 设备重启：目标设备重启后，工具无需人工干预，恢复通信后自动接上继续采集
- 优雅退出：手动中止后，`events.log` 记录退出事件，本机不留残留的 adb logcat 子进程
- 长时稳定：采集持续运行数小时后，心跳时间戳与当前时间差在数秒内，无长时间停滞

## 后续可能的扩展

- 若从「小时级会话」变为「长期常驻」：叠加系统级守护与日志切片轮转
- 若需要一次并采多台设备：由外层调度器批量拉起本工具实例，或将其改为多进程内建
- 若需要事中检索与告警：管道叠加结构化输出到本机日志聚合工具
